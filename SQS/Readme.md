## 1. What is AWS SQS?

**Amazon SQS (Simple Queue Service)** is a fully managed message queue service that enables decoupling of application components to build scalable, loosely coupled systems.

### Key Characteristics

- **Fully Managed** — No infrastructure to manage; AWS handles scaling, availability, and durability.
- **Message Queue** — Producers send messages; consumers retrieve and process them.
- **Asynchronous Processing** — Decouple components; consumer doesn't need to be available when message is sent.
- **Scalability** — Automatically scales to handle any message volume.
- **Durability** — Messages are redundantly stored across multiple AZs (99.999999999% durability).
- **Reliability** — At-least-once delivery guarantee (messages may be delivered more than once).
- **Low Latency** — High-performance message delivery.

### When to Use SQS

✓ Decoupling microservices
✓ Buffer spikes in traffic
✓ Long-running async tasks
✓ Batch processing jobs
✓ Task distribution across workers
✓ Retrying failed operations

## 2. Core Concepts

### 2.1 Producers and Consumers

```
Producer(s) → [SQS Queue] → Consumer(s)

- Producer: Application that sends messages to the queue
- Consumer: Application that retrieves and processes messages
- Both are decoupled — can operate independently
```

### 2.2 Messages

A **message** is the primary entity in SQS.

| Component | Description | Limit |
|-----------|-------------|-------|
| **Message ID** | Unique identifier (assigned by SQS) | Generated by AWS |
| **Body** | The message content | 1 byte - 256 KB |
| **Message Attributes** | Key-value metadata | Up to 10 attributes |
| **Receipt Handle** | Token to delete/modify the message | Generated on receipt |
| **Timestamp** | When message was sent | Assigned by AWS |
| **Visibility Timeout** | How long message hidden after receipt | 0 - 43,200 seconds |

### 2.3 Queue URLs

```
Standard Queue:
https://sqs.<region>.amazonaws.com/<account-id>/<queue-name>

Example:
https://sqs.us-east-1.amazonaws.com/123456789012/MyQueue
```

### 2.4 Message Retention

- **Default:** 4 days (345,600 seconds)
- **Range:** 1 minute - 14 days
- Messages deleted from queue after retention period expires
- Retention period doesn't affect visibility timeout

## 3. Queue Types

### 3.1 Standard Queues

```
Characteristics:
✓ Unlimited throughput
✓ Best-effort ordering (occasionally out of order)
✓ At-least-once delivery
✓ Lowest latency
✗ Messages may be delivered more than once

Use Cases:
- High-volume workloads
- Non-critical ordering requirements
- Asynchronous job processing
```

### 3.2 FIFO Queues (First-In-First-Out)

```
Characteristics:
✓ Guarantees order (strictly FIFO)
✓ Exactly-once processing
✓ Groups related messages together
✗ Limited throughput (300 msgs/sec by default, 3,000 with batching)
✗ Slightly higher latency

Queue Naming:
- Must end with ".fifo"
- Example: OrderQueue.fifo

Message Deduplication:
- Prevents duplicate messages in 5-minute window
- Deduplication ID or content-based deduplication

Message Grouping:
- Messages in same group are processed one at a time
- Different groups processed in parallel
```

### 3.3 Queue Type Comparison

| Feature | Standard | FIFO |
|---------|----------|------|
| Throughput | Unlimited | 300 msgs/sec (3,000 with batching) |
| Ordering | Best-effort | Guaranteed |
| Delivery | At-least-once | Exactly-once |
| Deduplication | None | Available (5-min window) |
| Message Groups | Not supported | Supported |
| Latency | Lowest | Higher |
| Cost | Lower | Slightly higher |

```python
    MaxNumberOfMessages=10,
    WaitTimeSeconds=20,  # Long polling
    MessageAttributeNames=['All']
)
```

### 7.2 Batch Receive Messages

```python
# Receive multiple messages
messages = client.receive_message(
    QueueUrl=queue_url,
    MaxNumberOfMessages=10,  # Max 10
    WaitTimeSeconds=20,
    MessageAttributeNames=['All']
)

for message in messages.get('Messages', []):
    process_message(message)
```

### 7.3 Long Polling vs Short Polling

```
Short Polling (WaitTimeSeconds=0):
- Returns immediately if no messages
- Higher CPU usage (frequent polls)
- Higher API calls = higher cost
- Lower latency but less efficient

Long Polling (WaitTimeSeconds=1-20):
- Waits up to specified seconds for messages
- Lower CPU usage
- Fewer API calls = lower cost
- Slight latency increase but efficient

Best Practice: Always use long polling (set WaitTimeSeconds=20)
```

### 8.1 Delete Message


### 8.2 Change Message Visibility (Extend/Reduce)

```python
# Extend visibility timeout (task taking longer)
client.change_message_visibility(
    QueueUrl=queue_url,
    ReceiptHandle=receipt_handle,
    VisibilityTimeoutInSeconds=600  # Extend by 10 minutes
)

# Reduce visibility timeout (make it available quickly)
client.change_message_visibility(
    QueueUrl=queue_url,
    ReceiptHandle=receipt_handle,
    VisibilityTimeoutInSeconds=0  # Immediately visible
)
```

### 8.3 Processing Pattern

```python
def process_messages(queue_url):
    while True:
        messages = client.receive_message(
            QueueUrl=queue_url,
            MaxNumberOfMessages=10,
            WaitTimeSeconds=20,
            MessageAttributeNames=['All']
        )

        if 'Messages' not in messages:
            continue

        for message in messages['Messages']:
            receipt_handle = message['ReceiptHandle']
            try:
                # Process message
                process_message(message)

                # Delete if successful
                client.delete_message(
                    QueueUrl=queue_url,
                    ReceiptHandle=receipt_handle
                )
            except Exception as e:
                print(f"Error processing message: {e}")
                # Don't delete - let visibility timeout expire
                # Message will reappear in queue for retry
```

### 8.4 Visibility Timeout Lifecycle

```
1. Message sent to queue (visible immediately)
2. Consumer receives message (becomes hidden)
3. VisibilityTimeout window starts
4. Consumer must delete before timeout expires
   - If deleted: Message removed from queue
   - If not deleted and timeout expires: Message becomes visible again
   - If processing fails during timeout: Message reprocessed by same/different consumer

Typical Timeout Strategy:
- VisibilityTimeout = Maximum expected processing time + buffer
- Example: Task takes 30 sec max → set to 60-90 seconds
- Use ChangeMessageVisibility if task takes longer than expected
```

## 9. Dead Letter Queues (DLQ)

### 9.1 What is a DLQ?

A **Dead Letter Queue** is a queue where messages that cannot be processed successfully are sent after a specified number of failed attempts.

```
Producer → [Main Queue] → Consumer → Success → Deleted
                            └─→ Failure (1st, 2nd... attempt)
                                └─→ [DLQ] (after max attempts)
```

### 9.2 Setting Up DLQ

```python
import json

# Create DLQ first
dlq_response = client.create_queue(
    QueueName='MyQueue-DLQ'
)
dlq_url = dlq_response['QueueUrl']

# Get DLQ ARN
dlq_attrs = client.get_queue_attributes(
    QueueUrl=dlq_url,
    AttributeNames=['QueueArn']
)
dlq_arn = dlq_attrs['Attributes']['QueueArn']

# Create main queue with redrive policy
redrive_policy = {
    'deadLetterTargetArn': dlq_arn,
    'maxReceiveCount': 3  # Move to DLQ after 3 failed attempts
}

client.create_queue(
    QueueName='MyQueue',
    Attributes={
        'RedrivePolicy': json.dumps(redrive_policy),
        'VisibilityTimeout': '300'
    }
)
```

### 9.3 Adding DLQ to Existing Queue

```python
import json

# Get main queue ARN
main_queue_attrs = client.get_queue_attributes(
    QueueUrl=main_queue_url,
    AttributeNames=['QueueArn']
)

# Get DLQ ARN
dlq_attrs = client.get_queue_attributes(
    QueueUrl=dlq_url,
    AttributeNames=['QueueArn']
)
dlq_arn = dlq_attrs['Attributes']['QueueArn']

# Set redrive policy
redrive_policy = {
    'deadLetterTargetArn': dlq_arn,
    'maxReceiveCount': 3
}

client.set_queue_attributes(
    QueueUrl=main_queue_url,
    Attributes={
        'RedrivePolicy': json.dumps(redrive_policy)
    }
)
```

### 9.4 Processing DLQ Messages

```python
def process_dlq_messages(dlq_url):
    """Process messages that failed too many times"""
    messages = client.receive_message(
        QueueUrl=dlq_url,
        MaxNumberOfMessages=10,
        WaitTimeSeconds=20
    )

    for message in messages.get('Messages', []):
        print(f"Failed message: {message['Body']}")
        print(f"ApproximateReceiveCount: {message['Attributes']['ApproximateReceiveCount']}")

        # Log for debugging/alerting
        log_failed_message(message)

        # Delete from DLQ
        client.delete_message(
            QueueUrl=dlq_url,
            ReceiptHandle=message['ReceiptHandle']
        )
```

### 9.5 DLQ vs Retry Policy

```
DLQ Use Cases:
✓ Messages that consistently fail
✓ Messages with invalid data
✓ Messages that require manual intervention
✓ Debugging and monitoring

Retry Policy (don't use DLQ immediately):
- Transient failures (network timeout, temporary db issue)
- Exponential backoff with max retries
- Then move to DLQ if still failing
```

## 13. Performance & Scaling

### 13.1 Throughput Limits

```
Standard Queues:
- Unlimited messages per second
- Unlimited throughput
- Scales automatically

FIFO Queues:
- 300 messages per second (default)
- With batch operations: 3,000 messages per second
- Exact once delivery limit

Request Quotas:
- SendMessage: per second per queue
- ReceiveMessage: per second per queue
- DeleteMessage: per second per queue
```

### 13.2 Scaling Strategies

```python
# 1. Scale consumers (horizontal scaling)
#    Add more consumer instances to process messages faster
#    Each consumer receives different messages

# 2. Batch operations
#    Process 10 messages per API call instead of 1
#    Reduces API calls and cost

# 3. Long polling
#    Set WaitTimeSeconds=20
#    Reduce API calls, lower cost

# 4. Increase visibility timeout gradually
#    Don't make it too long (waste resource if consumer fails)
#    Don't make it too short (reprocessing overhead)

# 5. Use FIFO with batching for ordered work
#    3,000 msgs/sec vs 300 msgs/sec = 10x throughput
```

## 14. Integration Patterns

### 14.1 SQS + Lambda

```python
# Lambda function triggered by SQS
def lambda_handler(event, context):
    for record in event['Records']:
        body = record['body']
        message_attributes = record['messageAttributes']

        # Process message
        process_message(body)

        # Lambda automatically deletes message if function succeeds
        # Returns with no error → message deleted
        # Raises exception → message returned to queue

    return {'statusCode': 200}
```

### 14.2 SQS + EC2 (Consumer)

```python
# Long-running consumer on EC2
while True:
    messages = sqs.receive_message(
        QueueUrl=queue_url,
        MaxNumberOfMessages=10,
        WaitTimeSeconds=20
    )

    for message in messages.get('Messages', []):
        try:
            process_message(message)
            sqs.delete_message(
                QueueUrl=queue_url,
                ReceiptHandle=message['ReceiptHandle']
            )
        except Exception:
            # Let visibility timeout handle retry
            pass
```

### 14.3 SQS + SNS (Fan-out Pattern)

```
SNS Topic → [SNS sends to] → SQS Queue 1
                              SQS Queue 2
                              SQS Queue 3
                              Email
                              Lambda

One message → Multiple consumers (decoupled)
```

### 14.4 Dead Letter Queue Pattern

```
Main Queue → Consumer → Success → Delete
                         Failure (1st try)
                         ↓ (message returns to queue)
                         Failure (2nd try)
                         ↓ (message returns again)
                         Failure (3rd try)
                         ↓ (maxReceiveCount=3)
                         DLQ → Log/Alert/Manual Review
```

```python
high_priority_queue = client.create_queue(

# Create standard queue
standard_queue = client.create_queue(
```

## 15. Cost Optimization

### 15.1 Pricing Model

```
Standard Queue:
- Free tier: 1 million requests per month
- After: $0.40 per 1 million requests
- 1 request = up to 64 KB

FIFO Queue:
- Free tier: 1 million requests per month
- After: $0.50 per 1 million requests
- Higher cost for exactly-once guarantee

Data Transfer:
- No charge for data in same region
- $0.02 per GB out of region
- $0.09 per GB out to internet
```

### 15.2 Cost Reduction Strategies

```
1. Use batch operations
   - 1 batch = 1 request
   - Send 10 messages at once = 1 request (not 10)
   - ~90% cost reduction

2. Use long polling
   - Set WaitTimeSeconds=20
   - Fewer poll requests
   - ~90% cost reduction vs short polling

3. Right-size visibility timeout
   - Don't set too high (waste processing time)
   - Don't set too low (reprocessing overhead)
   - Ideal: Max processing time + 50% buffer

4. Use FIFO only when needed
   - Slightly higher cost
   - Only for ordering requirements

5. Monitor and delete messages not needed
   - Don't keep old messages
   - Use message retention policy
   - Default 4 days is reasonable

6. Combine with SNS for fan-out
   - Instead of producer sending to 3 queues
   - Producer sends to SNS → SNS sends to 3 queues
   - Fewer API calls from producer
```

### 15.3 Cost Example

```
Scenario: Processing 3 million messages/day with batch operations

Without optimization:
- 3M messages × $0.40/1M = $1.20/day
- API calls per day = 3M (send) + 3M (receive) = 6M
- Cost = 6M × $0.40/1M = $2.40/day

With optimization (batch + long polling):
- Batch send: 3M / 10 = 300K batches = 1 request/batch = 300K API calls
- Long polling (20 sec wait): 86,400 seconds/day ÷ 20 = 4,320 polls/day
- Total API calls = 300K (send) + 4.3K (receive) = ~300K
- Cost = 300K × $0.40/1M = $0.12/day

Savings: ~95% reduction in API costs
```

## 16. Best Practices

### 16.1 General Best Practices

✓ **Always use long polling** (WaitTimeSeconds=20)
- Reduces API calls and cost
- Reduces latency in detecting new messages

✓ **Use batch operations** (SendMessageBatch, DeleteMessageBatch)
- Up to 10 messages per batch
- Reduces API calls, cost, and improves throughput

✓ **Set appropriate visibility timeout**
- Should be >= maximum expected processing time
- Use ChangeMessageVisibility if task takes longer

✓ **Implement proper error handling**
- Catch exceptions during message processing
- Don't delete message if processing fails
- Let visibility timeout handle retry

✓ **Use JSON for message body**
```python
import json
message = json.dumps({'user_id': 123, 'action': 'purchase'})
client.send_message(QueueUrl=queue_url, MessageBody=message)
```

✓ **Include message metadata**
- Use MessageAttributes for context
- Helps distinguish between message types

✓ **Monitor queue depth**
- Create alarms for ApproximateNumberOfMessagesVisible
- Monitor consumer health

✓ **Always implement DLQ for critical queues**
- Messages that fail consistently
- Prevents infinite retries

✓ **Log all processing failures**
- Track error patterns
- Debug issues quickly

✓ **Use message deduplication for FIFO**
- Prevents duplicate processing
- Exactly-once semantics

✓ **Scale consumers horizontally**
- Add more consumer instances
- Divide work across multiple processes

### 16.2 Security Best Practices

✓ **Enable encryption at rest**
- Use AWS managed keys or customer-managed keys

✓ **Use IAM policies** (principle of least privilege)
```json
{
  "Action": ["sqs:SendMessage"],
  "Resource": "arn:aws:sqs:*:*:QueueName"
}
```

✓ **Enable encryption in transit**
- Use HTTPS endpoints

✓ **Use VPC endpoints** for private access
- Avoid data traversing public internet

✓ **Encrypt sensitive data in message body**
- Even with SQS encryption

✓ **Audit queue access**
- Enable CloudTrail for SQS API calls

✓ **Set queue policy carefully**
- Use specific principals and actions
- Avoid wildcards

### 16.3 Operational Best Practices

✓ **Use queue naming conventions**
- dev-queue, prod-queue
- queue-name-purpose

✓ **Separate high-priority work**
- Create separate queues for different priorities
- Consumer prioritizes accordingly

✓ **Implement circuit breaker**
- Stop consuming if backend is down
- Prevent queue backlog

✓ **Use correlation IDs**
- Track message through system
- Useful for debugging

```python
import uuid

correlation_id = str(uuid.uuid4())
client.send_message(
    QueueUrl=queue_url,
    MessageBody='...',
    MessageAttributes={
        'CorrelationId': {
            'StringValue': correlation_id,
            'DataType': 'String'
        }
    }
)
```

✓ **Implement graceful shutdown**
- Finish processing current message
- Don't start new work
- Exit cleanly
